# QA Plan Workflow Enhancement — 2026-02-27

## 1. Background

### Problem Statement

The QA plan workflow produced documents with inconsistent formatting that made them difficult to read in Confluence. Specifically:

- Section headers used numeric prefixes at the top level (`## 2. QA Goals`) which conflicted with Confluence's own auto-numbering and made headings look redundant.
- Sub-categories (E2E, FUN, UX, etc.) within sections had no numeric ordering, making them hard to reference.
- Tables were overused — nearly every section had a table, including prose sections (QA Goals, Performance, Security) where a bullet list is cleaner and faster to scan.
- The QA Summary section in the plan was bloated with testing status sub-sections (E2E status, Performance results, Security checklist, etc.) that are only relevant AFTER testing is complete — at plan-stage, these are placeholder noise.
- QA Summary appeared mid-document rather than at the end, breaking the natural reading flow.
- After the reporter publishes its QA Summary to Confluence, the reporter was replacing the entire `QA Summary` section, discarding the Code Changes table the planner had already published.
- The `qa-plan-confluence-review` skill existed but was a copy of `qa-summary-review` — it was not tailored to QA plan pages at all.
- The feature-qa-planning workflow had no step to verify the Confluence page after publication.

## 2. Changes

### Change 1 — QA Plan Template Formatting (`qa-plan-architect-orchestrator/SKILL.md`)

**Rule: Main section headers — emoji prefix, no numeric prefix.**

All `##`-level section headers in the generated plan now use an emoji prefix with no numeric prefix:

```
## 📊 Summary        ✅
## 🎯 QA Goals       ✅
## 2. QA Goals       ❌ (removed)
```

**Rule: Sub-category and subsection headers — numbered.**

All `###`-level subsections within a section are now numbered with 1-based local numbers:

```
### 1. E2E: End to End    ✅
### 2. FUN: Functionality ✅
### E2E: End to End       ❌ (removed)
```

This applies to:
- Background subsections: `### 1. Key Problem Statement`, `### 2. Solution`, `### 3. Business Context`
- QA Goals sub-categories: `### 1. E2E` through `### 10. AUTO`
- QA Plan Test Key Points subsections: `### 1. UI Testing`, `### 2. Backend Testing`, etc.
- Risk & Mitigation subsections: `### 1. Technical Risks`, `### 2. Data Risks`, `### 3. UX Risks`
- Consolidated Reference Data: `### 1. Source Documents` through `### 4. Dependencies`
- Sign-off Checklist: `### 1. Development Team` through `### 5. Release Readiness`

**Rule: Tables restricted to specific sections only.**

| Section | Tables Allowed? |
|---------|-----------------|
| `## 📊 Summary` | ✅ Required (feature metadata table) |
| `## 🧪 Test Key Points` | ✅ Required (one table per subsection) |
| `## ⚠️ Risk & Mitigation` | ✅ Required (one table per subsection) |
| `## 📊 QA Summary` → `### 1. Code Changes` | ✅ Required (PR, Files Changed, PR Summary) |
| All other sections | ❌ Forbidden — bullets only |

Previously forbidden sections that had tables (now converted to bullet lists):
- QA Goals sub-categories
- Background / Business Context
- Consolidated Reference Data (Stakeholders, Dependencies)
- Sign-off Checklist

**Rule: QA Summary = Code Changes table only, placed at end.**

The plan-stage `## 📊 QA Summary` now contains a single subsection:

```markdown
## 📊 QA Summary

### 1. Code Changes

| PR | Files Changed | PR Summary |
|----|---------------|------------|
| [PR #NNN](url) | `path/to/file.ts` | Brief PR description |
```

Columns changed from `File Path / Change Type / Lines Changed / Risk Level / Test Status` → `PR / Files Changed / PR Summary` (PR-centric, not file-centric).

All other sub-sections that were previously in QA Summary (E2E Testing status, Performance, Security, Automation, etc.) have been removed from the plan stage. These are only relevant at reporting time and are now exclusively generated by the reporter agent.

The `## 📊 QA Summary` section is now positioned at the very end of the document — after all other sections including Notes. This ensures that when the Reporter agent appends sections 2-10, they naturally fall to the end of the page.

**Also updated:** `qa-plan-review/SKILL.md` structural checklist updated to enforce the new formatting rules.

### Change 2 — `qa-plan-confluence-review` Skill (full rewrite)

**File:** `workspace-planner/skills/qa-plan-confluence-review/SKILL.md`

Previously this file was a copy of `qa-summary-review` and was not tailored for QA plan pages.

The skill is now a full quality gate for published QA Plan Confluence pages, reviewing across three axes:

**Axis 1 — Formatting (F1–F6):**

| Check | What it verifies |
|-------|-----------------|
| F1 | Main section headers have emoji prefix, no numeric prefix |
| F2 | Subsections are numbered (1-based within each section) |
| F3 | Tables only in: Summary, Test Key Points, Risk & Mitigation, QA Summary Code Changes |
| F4 | QA Summary contains only `### 1. Code Changes` — no other subsections (Note: This strictly applies during the Plan Stage. In the QA Summary Stage, sections 2-10 will be appended) |
| F5 | QA Summary is positioned at the very end of the document (after all other sections, including Notes) |
| F6 | All Jira/PR references are hyperlinks |

**Axis 2 — Structure (S1–S5):**

| Check | What it verifies |
|-------|-----------------|
| S1 | All 9 required sections present and non-empty |
| S2 | QA Goals has all 10 sub-categories (1. E2E … 10. AUTO) |
| S3 | Test Key Points has at least one table with Priority / Test Key Points / Expected Results columns |
| S4 | Risk & Mitigation has at least one table with Risk / Impact / Likelihood / Mitigation columns |
| S5 | Summary table contains all required fields (Feature Link, Release Version, QA Owner, PR Link, etc.) |

**Axis 3 — Cross-Artifact Accuracy (A1–A4):**

| Check | What it verifies |
|-------|-----------------|
| A1 | Every Jira Acceptance Criterion has at least one test scenario in Test Key Points |
| A2 | Every PR in `github_pr.json` appears in QA Summary Code Changes with correct files |
| A3 | High-churn files (>50 lines changed) have a corresponding risk entry |
| A4 | If a Figma URL is present, UX goals and UI Testing subsection exist with content |

**Auto-fix vs. warn:** Formatting violations that require regeneration are flagged as FAIL (return to Phase 3). Confluence-side issues provide corrected text for the human. Cross-artifact gaps are warnings only.

**Output:** `projects/feature-plan/<feature-id>/qa_plan_confluence_review.md`

### Change 4 — Reporter QA Summary Generation (`workspace-reporter/skills/qa-summary/SKILL.md`)

**Problem:** The reporter was generating the full QA Summary from scratch, overwriting the Code Changes table that the planner had already published to Confluence.

**New behaviour — Append strategy:**

The reporter draft now contains **sections 2–10 only**. Section 1 (`Code Changes`) already exists on the Confluence page from the QA plan and is explicitly preserved during the Confluence merge.

**New section template:**

| Section | Type | Notes |
|---------|------|-------|
| `### 2. Defects Code Changes` | Table (new) | Defect-fixing PRs only (columns: Repository, PR, Defects Fixed, Risk Level, Notes) |
| `### 3. Overall QA Status` | Bullets | Risk level, defect counts, release recommendation |
| `### 4. Defect Status Summary` | Table | Open/Resolved/Won't Fix counts by priority |
| `### 5. Resolved Defects Detail` | Table (P0/P1 only) | Linked Jira IDs, resolution, notes |
| `### 6. Test Coverage` | Bullets | |
| `### 7. Performance` | Bullets | |
| `### 8. Security / Compliance` | Bullets | |
| `### 9. Regression Testing` | Bullets | |
| `### 10. Automation Coverage` | Bullets | |

**Confluence Merge Strategy (Phase 5 of QA Summary workflow):**

```
1. Locate the `## 📊 QA Summary` section at the very end of the Confluence page
2. Preserve existing Code Changes subsection (section 1 from plan)
3. Append sections 2–10 from draft immediately after the Code Changes subsection
4. Do not duplicate any subsection already present
```

The `Section 2. Defects Code Changes` table uses columns:
`Repository | PR | Defects Fixed | Risk Level | Notes`
where `Defects Fixed` contains hyperlinked Jira IDs of bugs resolved by the PR.

#### Pending 2e — Wait & Merge Semantics

1. **Launch:** Invoke all selected skills in parallel (same turn; multiple spawn calls).
2. **Completion detection:**
   - **sessions_spawn:** Poll `subagents list` until all spawned agents report done, or check output files exist.
   - **mcp_task:** Each call returns when the subagent completes. Launch with `run_in_background: false` to block per-task, or use `run_in_background: true` and poll for output files.
3. **Merge trigger:** Only when at least `qa-plan-atlassian` output exists (required). Optional outputs (figma, github) may be absent.
4. **qa-plan-synthesize input:** Pass the list of existing domain summary paths. Synthesize generates the comprehensive Test Key Points table (mapping ACs to Code Changes), merges available domain summaries into the 9-section template, and fills gaps with placeholders where a source was skipped.

#### Pending 2f — Error Handling
