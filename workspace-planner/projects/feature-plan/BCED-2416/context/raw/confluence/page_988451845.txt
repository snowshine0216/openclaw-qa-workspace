1. Problem statement

There are plenty of graph formatting properties available in Developer but not
available in Workstation yet. The gap exists because Developer calls COM APIs to
set these graph properties, while Workstation calls XML APIs to set the graph
properties. However, there's a gap between COM APIs and XML APIs. Some COM APIs
don't have their XML API counterparts. Moreover, configuring graph properties
from Developer and from Workstation (Authoring Web) may also follow different
workflows.
This feature aims at doing research on the gap of supporting rendering and
configuring graph formatting properties by XML API, and figure out how the gap
can be filled in and how much effort is needed.
Moreover, if we have enough capacity to implement the unsupported XML APIs, we
should also include the implementation as part of this feature scope.


2. Feature Summary

The following graph properties (inherited from the property list in TH4202,
except for I16507 which was already supported in 11.2 RC) are in the scope of
this feature.

Note:

Current workflow on developer:

The developer calls the following function to put the picture as the background
for option "Texture"

In this function, we can

 1. directly use the picture data if the ippPicture variant has type (VT_ARRAY |
    VT_UI1)
 2. load the file from the file system using the iName (which is the address of
    the file)

(The CDSSGraphFill is a member object of CDSSGraphStyle as mBGFill)

Solutions:

(1) A candidate solution is that we still use
the DssRWManipulationSetGraphProperties manipulation but with the original value
"v" for iName (or ippPicture) and a new parameter for ippPicture (or iName)

(2) Another solution is that we can use a new RSD manipulation for this
specifical graph-related property

Solution 1

Pros:

 1. we can use the current graph manipulation workflow, so no need to implement
    from scratch

Cons:

 1. since for this graph manipulation, we need more information about the
    flip-type, the scale type, and the format, we need the client to embed more
    attributes and values in the <gp> node. And on the server-side, we need a
    new workflow to parse these parameters
 2. since every time change a graph's property, all the previous graph
    manipulations will be auto-applied again. This property is a little bit
    different compared with others, because we may need to load the image from
    local disk, which might be time-consuming compared with other manipulations

Solution 2

Pros

 1. no need to apply this manipulation every time the graph property is modified

Cons:

 1. need more code change to implement from scratch
 2. we still need to put this manipulation into graph delta, so it will not be
    missed when re-applying all graph manipulation.

Generally, I think the first solution is better.
