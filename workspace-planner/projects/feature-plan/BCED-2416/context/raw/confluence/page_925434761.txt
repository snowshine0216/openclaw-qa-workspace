Library Desktop is a desktop app for Mac and Windows that renders the
MicroStrategy Library for data consumption with an instant response user
experience.



These are five major areas of development in the roadmap to the Library Desktop
MVP:

 1. Functionality parity with Library Web using a desktop window paradigm as
    opposed to a web page paradigm.

 2. Caching is used to download dossier data for fast consumption.

 3. Authentication in Library Desktop remembers the user credentials.

 4. Workstation Integration. Launch the Library Desktop from Workstation.

 5. Offline capability in Library Desktop.

Architecture

--------------------------------------------------------------------------------

Library Desktop is built on top of Electron technology which is a popular
framework used by well known desktop apps.

→ Electron employs a Chromium based browser which allows us to port the user
interface from the Library Web app to the Library Desktop.

→ Electron supports distributions on Mac and Windows which facilitates the
portability in multiple operating systems.

https://microstrategy.atlassian.net/wiki/x/2wIpMQ
[https://microstrategy.atlassian.net/wiki/x/2wIpMQ]



There are a handful of invariants that are established for the Library Desktop:

 1. Library Desktop can have 0, 1, or more Environments, where an environment is
    the URL of a Library Web Server.

 2. Library Desktop has one Active Environment when more than one environments
    are set. The user selects the Active Environment. If the user does not have
    a chance to select an active environment, then the app selects the first one
    as default.

 3. Library Desktop employs to first class non-modal windows to display the user
    interface:
    
    1. Library Window displays the user’s library of the Active Environment.
       There is only one Library Window instance at any time in the applications
       lifetime. The Library Window may be hidden or closed.
    
    2. Dossier Window displays the contents of a dossier that is selected from
       the Library Window. There can be 0 or many Dossier Window instances
       opened at any time in the application’s lifetime.

 4. Switching the Active Environment in the application closes all Dossier
    Windows and updates the contents of the Library Window to display the user
    library from the new selected environment.

 5. Library Desktop employs other second class windows for example for Login and
    Preferences which function as modal windows.

In Mac, the Library Window can be closed while the app remains opened. The user
can re-open the window from the menu bar. This is a standard paradigm of the Mac
OS.



The following sections describe architectural concerns that are being considered
in the app.


Packaging and Functionality

Library Desktop is packaged as an Electron app and uses React to render the user
interface. The user interface is ported from the Library Web app but rendered
locally in the desktop app (without using Web View techniques). End user actions
produce manipulations that are executed through the Rest API in the Library Web
Server.


How is the application packaged

The application is packaged as a standard Electron app using Electron Builder.
Electron Builder produces installers for both Windows and Mac.


How is the application versioned

The application follows the latest version of the platform so Library Desktop
ships a GA version on every quarter. Customers see a single line of Library
Desktop releases and simply update to the latest version of the app.

→ Changes in versions that are shipped as GA (eg. 11.2.4 or 11.3.4) must be
included in the next production release as well.

→ Library Desktop can connect to older Library Web Server environments. If a
customer updates Library Desktop from 11.3.3 to 11.4, the customer can connect
Library Desktop 11.4 to the 11.3.x Library Web environment.

→ Library Desktop packages the JS bundles and static assets of the latest
release and renders the UI with the packaged bundles. Therefore the application
must be backwards compatible when connected to older Library Web servers.

How is the application patched

We don’t expect to patch the Library Desktop to individual customers. Instead,
patches are provided as minor updates in the Download Site and Stores and all
customers can upgrade the latest version.




How is the application uninstalled

On Mac an uninstaller should be provided to uninstall the application. On
Windows the application uninstalls using the Windows Add and Remove Programs
tool.

Uninstalling the application should delete the user's data. Specifically, the
dossier cached data.




How is the user interface rendered in Library Desktop

The React based applications from Library Web are ported to Library Desktop. The
React applications render the user interface for the Library Window and the
Dossier Window. However, the React applications are rendered from a local file
instead of a web server so special considerations are required for cookie
management.




How does Library Desktop ensure rendering parity with Library Web

The Library Desktop is built on top of the Chromium Browser so rendering should
be equivalent to Library Web in Chrome browsers. Customers using Library Web in
IE and Library Desktop (in Chromium) will observer the same differences as
between Library IE and Chrome browsers.

Caching and Offline

Library Desktop uses a Background Cache Process to store the dossier content in
a database. The database can then be used to provide the dossier content without
a network roundtrip.

What data is stored in the database

The database stores JSON originating from the Library Web Server. The database
stores three classes of JSON data:

 * Library JSON stores the contents of the user’s library.

 * Dossier JSON stores the contents of dossiers.

 * Document JSON stores the contents of RSD documents.




What is the database schema

The goal of the schema design is to optimize for Read operations which must
return as fast as possible to achieve sub-second response time when opening
dossiers. Write operations are less critical as the application’s Background
Cache runs in the background when the application is idle. We expect to design
an schema that matches the data requests coming from the user interface. In
general, we anticipate the following stores:

 * Library - stores the library contents.

 * Dossier - stores high level dossier data (e.g. name, last-update,
   table-of-contents).

 * Page - stores dossier page data.

 * Grid - stores grid data for incremental fetching.

 * Document - stores report-services-document data.

When are dossiers removed from the database after removing from the Library

We remove dossiers from the database after the user removes a dossier from the
library. We expect to have a grace period before removing the dossier from the
database to give users a chance to bring back a dossier to the library before
losing the cache.

A dossier can be removed from Workstation or Library Web and Mobile clients. In
this case dossiers are removed from the cache when launching Library Desktop.

Authentication

Library Desktop supports the authentication modes of the Library Web. In
addition, Library Desktop stores the user credentials in the desktop.


How is the application storing the user’s credentials

We plan to use the Keytar library to store user credentials in the OS.

https://github.com/atom/node-keytar [https://github.com/atom/node-keytar]

https://github.com/atom/node-keytar/blob/master/LICENSE.md
[https://github.com/atom/node-keytar/blob/master/LICENSE.md]

4 incomplete What is the expectation of storing credential for each
authentication mode.




What is the support to store credentials for Mac and Windows

On Mac, credentials are stored in Key Chain. On Windows, credentials are stored
in the Credential Vault.




How can the credentials be flushed

The Keytar library provides a way to delete credentials so we can potentially
implement a “Forget” option.

https://github.com/atom/node-keytar#deletepasswordservice-account
[https://github.com/atom/node-keytar#deletepasswordservice-account]




How will the background cache mechanism authenticate on behalf of the user

The Library Desktop authenticates the user on each environment for which the
credentials can be obtained from the Keytar library. Library Desktop, then uses
the session on each environment to execute the Background Cache mechanism.




What is the Application Type of Library Desktop

Library Desktop will likely introduce a new application type for Login. Platform
Analytics should be able to break down activity for the Library Desktop.
