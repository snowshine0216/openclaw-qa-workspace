1. Purpose

Currently, when users are creating RSDs, they generally do so in MicroStrategy
Web, and this often becomes a fragmented experience, as users can do a lot of
formatting in Web, but then have to switch to Developer to complete a lot of
common tasks. Configuring advanced grid options is one of the tasks that force
users to switch to the Developer to achieve the task.


2. Design

Overall, we are taking many of the commonly used and high-value formatting
options over from Developer into Workstation.

(ZH: I think it is not only web editor, but also workstation, right?)(P3)

(FL: I have updated to only include the Workstation)


2.1 Components

This feature is mainly about to migrate the settings of Documents like Grid Data
Options from Developer to Web, which needs changes on both Server side and
Client side.

To be specific, the Server side will investigate the COM API and figure out the
corresponding property/object. Then find out whether is a matching XML API
available to do the same work. 

2.1.1 INVESTIGATION OF COM APIS

The detailed investigation result of the mapping between COM API and XML API can
be found at the confluence page here Developer property parity
[https://microstrategy.atlassian.net/wiki/spaces/TECTools/pages/745277867/Developer+property+parity]

When processing this manipulation, we will

 1. clear the current CustomSubtotals collection in the template.
 2. find the Subtotal definition from subtotal type
 3. check whether the Subtotal has already in the TemplateSubtotals collection,
    if not, create a new one.
 4. create a new CustomSubtotal from the Subtotal definition and add it to the
    CustomSubtotals collection
 5. add the units in the unit collection to the CustomSubtotal

If no units in the unit collection, then it's Grand total, we will create the
CustomSubtotal object without adding the units to it.

CUSTOM SUBTOTAL

If a new custom subtotal is defined, it will be added to the IDSSTemplate's
Subtotals collection. We need a new manipulation to do this, where we let the
user define different subtotal on each metric.

(ZH: Can you help describe the detailed steps for this manipulation? Is the
custom subtotal a first class object?)(P2)

(FL: I have added the steps for handle manipulation; custom subtotal is actually
a TemplateSubtotal, so it isn't a first class object)

(ZH: How can a user remove/clear/modify custom subtotal?)(P2)

(FL: I have added the description of how users can remove/clear/modify the
custom subtotal)

For this manipulation, we will create a new TemplateSubtotal object and add it
to the Template's TemplateSubtotals collection. And for each unit (metric), we
will create a MetricSubtotal object and put them in the newly created
TemplateSubtotal.

If the user wants to modify the custom subtotal, we could always delete the
original ones and create new subtotals. so:

 1. To remove all the custom totals, the client could send this XML without name
    and units collection, thus all the custom totals will be cleared.
 2. To remove some of the custom totals, we can clear the TemplateSubtotals
    collection first and re-create Subtotals to be kept
 3. To modify some custom total, we can re-create it from scratch

AXES LABEL ROTATION

YellowEnhance

This setting can be changed via <rbd> command, but the server side needs to
enhance the graph-related workflow. 

Currently, clients can change graph style by sending <rbd> command with
manipulation 37 (DssRWManipulationSetGraphProperties), and when the server
receives these manipulations, we add all this kind of manipulations to the graph
object, and they will be applied when saving the document to XML.

However, in the current workflow, when applying these manipulations in
GraphUtil.cpp:ApplyManipulationXML3, we will only process limited graph
properties and skip others. For instance, the Label Alignment can be applied
there, but Label Rotation is skipped.

So in order to support this setting, we need to add this property
in GraphUtil.cpp:ApplyManipulationXML3, so this manipulation can be applied to
the graph successfully.

3. Non-functional requirements (NFRs)
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements]


3.1 Security
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements]

For this feature, we just expose existing settings to Workstation other than
Developer via XML command. 

 1. since these settings have already been used in Developer, so let users set
    them from other clients will not bring any risk.
 2. we want to achieve the goal by introducing new manipulation or enhancing
    existing ones, so no extra security issues will be added.


3.2 Scalability
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements]
(including universal benchmark)

No performance and scalability degradation on existing functionality. 


3.3 Performance
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements] 

No performance and scalability degradation on existing functionality. 


3.4 Error and exception handling
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements]

Since this feature is implemented based on the current document manipulation
workflow, errors and exceptions may occur during:

 1. parsing XML command if some tag or attribute is missing or have an invalid
    value
 2. applying manipulation, for example, cannot find some object or change its
    property value

Once error and exception happen, we will comply with the current workflow,
return the error to the caller function and let it process the error.


3.5 Logging
[https://github.microstrategy.com/pages/kiai/developers-guide/#Logging]

Since this feature is implemented based on the current document manipulation
workflow, we will use the existing logging measures when an error or exception
happens.


3.6 Monitoring
[https://github.microstrategy.com/pages/kiai/developers-guide/#Monitoring] &
Telemetry

Since this feature is implemented based on the current document manipulation
workflow, the monitoring and telemetry will keep their original behavior.


3.7 Stability

This feature ideally will not bring any stability issues to the server. 


3.8 Internationalization
[https://github.microstrategy.com/pages/kiai/developers-guide/#Internationalization]

The error message's locale return to the client will be automatically adjusted
on the user's language preferences.


3.9 Accessibility
[https://github.microstrategy.com/pages/kiai/developers-guide/#Accessibility]

NA (This feature only focus on server-side changes)


3.10 Compatibility
[https://github.microstrategy.com/pages/kiai/developers-guide/#Non-functional-Requirements]

For this feature, we will add some new report manipulations, and they will not
affect existing behavior.

At the same time, we will enhance some manipulations to let them be able to
handle more complicated scenarios without causing regression issues.


3.11 Feature Flag

NA


3.12 Administration and Configuration

This feature only exposes some settings via XML command, which will not change
anything about administration or configuration.
