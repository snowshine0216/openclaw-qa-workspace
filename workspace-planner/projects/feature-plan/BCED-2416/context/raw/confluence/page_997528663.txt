1. Introduction

A prompt is a question the system presents to a user when a report is executed.
How the user answers the question determines what data is displayed on the
report when it is returned from your data source.

A report designer can include one or more prompts in any report. Prompts are an
effective tool to:

 * Allow each user who executes the report to request individualized sets of
   data from your data source when they answer the prompts and runs the report.
   Effectively, each user creates a filter for the report.

 * Allow the report designer to create a smaller number of reports overall,
   using more inclusive objects, rather than having to create numerous, more
   specific reports that are individualized to each analyst.

 * Allow the report designer to ensure that the objects on a report are the
   latest available objects in the project. This is possible using a search
   object in a prompt. When a user launches a prompt by running a report, the
   search object retrieves the latest objects that fit the search criteria the
   report designer defined. Thus, no matter when the prompt was created, each
   time a user executes the report, the user chooses prompt answers from a list
   of the most up-to-date objects available in the project, including objects
   that may not have existed when the prompt was created.

 * Allow users to keep the objects on their saved reports up-to-date. Users can
   save a prompted report so that the objects within the prompt remain connected
   to the original objects within the project that they were originally based on
   when the prompt was created. If objects are modified or deleted in the
   project, the report can reflect those changes the next time the prompted
   report is run.


1.1 Components of a Prompt

The pieces that make up a prompt control how a prompt appears and how it
functions. These components include the following:

 * Answer requirement: This component lets you determine whether users are
   required to answer the prompt or an answer is optional. If an answer is
   required, a report cannot be executed until an answer is provided.

 * Default prompt answers: This component lets you include a pre-selected answer
   for the prompt, which the user can then accept, replace with a different
   answer, or accept and add more answers.

 * Title and description: This component lets you provide a useful name and
   description for the prompt, which can significantly impact whether a user
   finds prompts to be simple or complex.

 * Style: This component determines the appearance of the prompt and the layout
   of how users select answers for the prompt.

Consider your users' needs and the purpose of the report and the objects on it
when you decide on these options. If you plan to apply a schedule to a prompted
report, the decisions you make about answer requirements and default answers
will affect how the report is filtered when it is automatically executed on
schedule.


1.2 Stand-Alone Prompt vs. Prompt as Part of a Report or Filter

A stand-alone prompt is a prompt that is created as an independent MicroStrategy
object. A stand-alone prompt can then be used on many different reports, as well
as on filters, metrics, and other objects, and can be used by other report
designers. A stand-alone prompt gives report designers flexibility.

Prompts can also be created as intrinsic part of a given report, filter, or
custom group, at the same time the report, filter, or custom group itself is
being created. Prompts created as part of another object are saved with the
definition of that object. Therefore, a prompt created as part of another object
cannot be used on any other project.


1.3 Comparing a Prompt and Filter

A prompt is similar to a filter because a prompt determines the specific data to
be displayed on a report. The difference is that you create a filter for a
report to provide a single, specific definition for the report. A filtered
report then displays the same set of data to every user who executes that
report. In contrast, a prompt dynamically modifies the contents of a report
based on the user’s answers when the user executes the report. With prompts,
users can determine the objects (attributes, attribute elements, metrics, and so
on) that they want to be part of, or excluded from, the report query that is
sent to the data source. Therefore, a prompt can be seen as a way for each user
to create their filter for a given report. For example:

 * Users can choose from several existing filters to determine exactly which
   filter screens the data on the report that they are about to execute. To
   achieve this, create an object prompt that contains existing filters and
   place that object prompt on a report. Then the user can choose which filter
   to apply to the report.

 * Prompts allow a report to have a dynamic report definition, which users can
   change each time that they submit the query by choosing different answers
   when prompted. If you create an attribute prompt containing the Year
   attribute, users are prompted to select the year for which they want the
   report results. The report can be run the first time by selecting 2005 and
   then a second time by selecting 2006.


1.4 Types of Prompts

The correct prompt type to create depends on what report objects you want users
to be able to base a filter on to filter report data.

Prompt Type

Prompt Name

What Data it Allows Users to Define

Filter definition prompts

This type of prompt allows users to determine how the report’s data is filtered,
based on attributes in a hierarchy, lists of attribute elements, or metrics.

Attribute Element List Prompt
Prompts users to choose from an attribute element list

Users can select prompt answers from a limited list of specific attribute
elements. This prompt is the most restrictive of the filter definition prompts,
because the user has the least number of attribute elements to select answers
from.

Attribute Qualification Prompt
Prompts users to qualify on an attribute

Users can select prompt answers from a list of attribute elements from a single
attribute. This prompt is more restrictive than the hierarchy qualification
prompt, because the user has fewer attribute elements to select answers from.

Metric Qualification Prompt
Prompts users to qualify on a metric

Users can define a metric qualification, which determines what data should be
displayed for one or more specific metrics on the report.

Hierarchy Qualification Prompt
Prompts users to choose from all attributes in a hierarchy

Users can select prompt answers from one or more attribute elements from one or
more attributes. The attribute elements that they select are used to filter data
displayed on the report. This prompt lets you give users the largest number of
attribute elements to choose from when they answer the prompt to define their
filtering criteria.

For example, on a report displaying profit forecasts, if the prompt lets users
select from the Product hierarchy, one user might choose to see forecasts for
certain electronic products, while another user might select different
electronics products, or all media products.

Object Prompts

This type of prompt allows users to select objects to include in a report, such
as attributes, metrics, or filters

Object Prompt

Users can use this prompt to add more data to a report. Users select objects
(such as attributes or metrics) that they want to add to the report. Users can
also choose from among a selection of filters, to apply a filter that is most
useful for their analysis purposes.

Value Prompts

This prompt type lets users select a single value, such as a date or a sepcific
text string, and filter report data based on their selection.

Date & Time Prompt

Users select a specific data and time for which to see data. This prompt is used
in a filter.

Numeric Prompt

Users enter a specific number, up to 15 digits, which is then used as part of a
filter or whithin a metric, to look for specific numeric data.

Note, if a user enters more than 15 digits for a numeric prompt, the data is
converted to scientific notation. If precision is needed beyond 15 digits, you
should use a Big Decimal value prompt instead.

Text Prompt

Users enter a word or phrase, which is then used as part of a filter to look for
specific data with that text.

Big Decimal Prompt

Users can enter up to 38 digits, to search for numeric data with the Big Decimal
data type assigned to it.

The above prompts and Level prompts are supported in Library Web. Level prompts
allow users to specify the level of calculation for a metric. Level prompts
cannot be created from MicroStrategy Web, they must be created in MicroStrategy
Developer. And in Web-Server, we call Level Prompt as WebDimtyPrompt

BTW, when we create a new Report and choose "Report Builder", it will also open
the Prompt page to create the report, so when fixing Prompt related issues, we
may also need to cover this case.

2.2 JUIL on Client side

The Client side part is based on MVC architecture with JS. The JS Library is
under the namespace mstr, and located in javascript/mstr folder. We have below
major components:

 * Model Layer:
   
   * mstr.models: maintains some properties data internally, provides commands
     for manipulating that data, raise events when its data is modified,
     provides methods for adding/removing event listeners to the Model.

 * View Layer:
   
   * mstr.views: maintains a reference to a Model, provides methods for
     rendering its HTML GUI, and for updating that GUI in response to events.
   
   * mstr.itemrenderers: corresponds to the itemRendererClass property of
     related block files in Web-Server side. For example, <property
     name="itemRendererClass">mstr.itemrenderers.CheckList</property>. It is a
     javascript object responsible of rendering data items in the view's GUI.
     For example, a ListView uses an item renderer to render list items.
   
   * mstr.behaviors: setting the properties in layout files in Web-Server side,
     like on_set_enabled, on_set_visible. Note there is an exception for
     mstr.behaviors.ExpandableTreeView, this should be a part of
     mstr.itemrenderers and is called in TreeView.xml. It should be a mistake by
     unknown historical reasons.

 * Controller Layer:
   
   * mstr.controllers: contains two parts: Factory and EventManager, Factory is
     a singleton that maintains a collection of model and viewer objects.
     Factory provides methods for creating, destroying and referencing those
     objects. EventManager is a singleton with methods and members for managing
     propagation of events among objects.

 * Others:
   
   * mstr.http: helper methods for handling Ajax request, mainly includes
     FrameProcessor and XHRProcessor.
   
   * mstr.utils: helper methods for JavaScript object data structure and DOM
     manipulations.
   
   * mstr.bone.editors: renders the tabs of "Properties and Formatting" editor
     and "Document Properties" editor of Document on the GUI.
   
   * mstr.lang: helper methods for HTML events and MSTR events.
   
   * mstr.xml: helper methods for XML manipulations.

Note that each component is not centralized at a single place, instead each of
the package are distributed across multiple files. (due to code refactoring to
improve performance, so that only required code will be loaded)

3.3.1.1 COMPLEX PROMPT ANSWER

When we open a Prompt page, for most cases, we can only see one area for a
prompt question like below-left which is called Simple Prompt Answer. But for
some cases, we may also see another area for the default selection like
below-right which is called Complex Prompt Answer (it is always displayed as
static text on the GUI):

The reason is, when we render a Prompt Question, we will always try to render
the default prompt answer together. For most cases, we can render the default
prompt answer in the normal area, but for some cases, we are not able to render
in the normal area:

 * Default answer’s elements are not available any more for the prompt. For
   example, in the past we choose 2010 as default answer for Year, but later
   something changes and Year attribute doesn’t have 2020 element any more. Then
   we will display 2010 in default selection area.

 * In the past the prompt's style supports multiple-selection like checkbox, and
   we have 2009 and 2010 as default answer. Then we change the prompt's style to
   single-selection like radio-button. Then we will display 2009 and 2010 in
   default selection area.

 * The prompt's default answer's datatype is unknown, or not match prompt's
   requirement for datatype. For example, the answer is a Date but prompt
   requires a Integer. Then we will also display the default answer in default
   selection area.

 * The default answer’s expression is too complicate, like Dynamic Date in the
   above image, we are not able to display it in Shopping Cart, then we will
   also display it in default selection area.

There are also some other cases, and we can find the detailed logic in
isAnswerComplex() method in our Java code library. Note that there are multiple
isAnswerComplex() methods according to multiple prompt types and styles.

Since complex prompt answer is just a static text, if user choose this as the
final answer, the workflow will be quite simple, we just send back the text back
to Web Server.

3.3.1.2 SIMPLE PROMPT ANSWER

There are three kinds of prompt answers that inherit from
mstr.models.PromptAnswer: Constant Prompt Answer, List Prompt Answer and
Expression Prompt Answer.

3.3.1.2.1 CONSTANT PROMPT ANSWER

Constant Prompt Answer is an answer to a Constant Prompt (mostly a Value
Prompt), its value is a string, and we can use the "min", "max" and "dataType"
properties to restrict the string’s content.

Note that when we validate Constant Prompt Answer, we will first convert it to
an array using specific delimiters and verify each element, then convert it back
to string as new formatted value. The reason is, Value Prompt may also have
multiSelect property with true, then the string will be a delimited list and we
need to validate each sub-element, for example, Free-Form SQL case.

3.3.1.2.2 LIST PROMPT ANSWER

List Prompt Answer is an answer which’s value is specified by a list of objects
(e.g., attributes, metrics, etc), and we can use the optional "min" and "max"
properties define further restrictions on the end-user’s selections. It is
usually used in Attribute Element List Prompt and Object Prompt.

There are three different ways to get List Prompt Answer depends on prompt’s
style:

 * Tree/Shopping-Cart: We will get answer from the items of its selected
   property: this.props['selected'].props['items']

 * Pulldown: We will get answer from the items of its available property based
   on the selectedIndices property.

 * Other styles: We will get answer from its preSelectedItems property. This
   property was introduced to remember previous selections of Prompts for the
   Search cases. In the past, our prompt answers were only based on current
   elements on the GUI, for example, for Category, if we search ‘o' and then
   select ‘Book’, then search ‘u’, then search ‘o’ again, ‘Book’ will be not
   selected. The reason is, when we search ‘u’, ‘Book’ is not displayed on the
   GUI, then our selection will remove this, then when we search ‘o’ again, we
   will not consider 'Book’ as selected.
   However, this new property makes our logic much more complex and confused, we
   need to maintain this property for all the manipulations that will change our
   prompt answers. So we need to pay much more attention when we meet issues
   about this part.

3.3.1.2.3 EXPRESSION PROMPT ANSWER

Expression Prompt Answer is an answer which’s value is specified by an
expression tree. This tree is maintained in the "selected" property.
Additionally, the "available" property specifies a list of objects which the
end-user may use in the expression tree. Also, the optional "min" and "max"
properties define further restrictions on the end-user’s selections. It is
usually used in Attribute/Metric/Hierarchy Qualification Prompt.

There are two different ways to get Expression Prompt Answer

 * DualExpressionListCart: For most cases, we will not use it, it is only
   available when we are using SAP attribute in Prompt and there is no default
   function for its expression (Refer to
   PromptExpressionWidgetTransform.java::shouldRenderDualCart()). Its answer
   depends on the selectedIncludes property and selectedExcludes property in
   DELC.prototype.buildSelectedModel()

 * Other cases: We will get answer from its selected property. Since we are
   using expression as answer’s format, there are lots of models for the parts
   of expression, like IncludeListModel, ExprTypeListModel, BranchQualListModel
   and so on. When we do any operations on the GUI, we will call related models'
   update methods and update the prompt’s selected property. We can find all
   these in mstr.models.ExpressionPromptAnswer

3.3.3.2 PROMPT ANSWER MODES

This parameter indicates how prompts should be answered, is a
parameter(promptAnswerMode) of prompt answer events. The following values are
available from EnumWebPromptAnswerMode:

 * DO_NOT_PASS_PROMPT_ANSWERS = 0
   Do nothing to the prompts, just display them on the GUI, even we have prompt
   answers in the event, those answers will be ignored.

 * ALL_PROMPTS_USE_DEFAULT_ANSWER = 1

 * OPTIONAL_PROMPTS_USE_EMPTY_ANSWER = 2

 * USE_SOURCE_ANSWERS_FOR_PROMPT_IN_PROMP = 4
   This mode is only used for the case we have both linkAnswers and srcAnswers,
   and we will use their values to answer target’s prompts.

Note there is no default value for this parameter, if it is unset, then we will
use the value of prompt answer parameters in the event.

3.3.3.3 LINK ANSWER MODES

This parameter is only for Link Drill, and is not a parameter of events, we can
get it from linkAnswers XML. so it will only take effect when linkAnswers is
used. The following values are available from EnumLinkAnswerMode:

 * SAME_PROMPT = 1, answer with the same prompt from the source

 * DO_NOT_ANSWER = 2, don’t auto-answer prompts, always display prompt page.

 * CLOSE = 3, answer with an empty answer

 * DYNAMIC = 4, answer dynamically with the selections in source, for example,
   the selector value, the chosen cell in grid.

 * STATIC = 5, answer with a static list of elements

 * CURRENT_UNIT = 6, answer using current unit

 * ALL_VALID_UNITS = 7, answer using all valid units

 * USE_DEFAULT_ANSWER = 8, use default answer

 * DYNAMIC_OR_SAME_PROMPT = 9, answer dynamically, or with prompt from the
   source

Detailed workflow can be found in WebHyperLinkAnswerImpl.answerPrompt()

On Client side, the biggest problem is the big JS files and mixed components in
each files.

 * In common.js file (10180 lines), we have  models, views, behaviors,
   itemrenderers, http

 * In core.js file (9047 lines), we have models, controllers, behaviors, utils,
   http

 * In prompts.js file (7698 lines), we have models, views, itemrenderers,
   behaviors, utils, http

 * In toolbarsAndAnnotations.js file (11717 lines), we have models, views,

 * In trees.js file (5861 lines), we have models, views, behaviors, utils

 * In expressions.js file (10674 lines), we have models, views, utils

This is quite unfriendly, and will make engineers really confused. After so many
years' code change, now these related file names also doesn't make sense because
when we adding new methods, it is not easy to identify which file to be added.
And when we change some methods, sometimes we may consider the changed method
only be used for some scenarios, but in fact it may be a public method.

So below is the enhance plan:


5.1 Move methods of same layer together (Optional): 80 points

In fact, we already have many empty files for these layers under the same "mstr"
folder, like models.js, views.js, controllers.js. It seems that in the past, we
already had the plan to move methods, but it was suspended due to some reasons.

So here we can continue this suspended work, instead of moving
all mstr.models.XXX methods to models.js, for each object, we will create a
standalone file with the same name like TextBox.js, and put these files under
related layer’s folder.

Risk: Medium

 1. We need to check the dependency and requirements for these files, make sure
    all the files that depend on these files will be changed to new files.

 2. For Library, WorkStation and any other products that will require these
    files, we need to notice them to change related scripts to require new
    files.

 3. For some plugins that overrides methods of these files, we need to make sure
    they still work. We can write a new plugin to test this case to make sure
    this change will not affect customer's plugins.


5.2 Remove useless codes for old browsers: 24 points

In the past, we added many codes to handle specific browser cases (especially
for IE). And now those cases are not valid any more since we don’t support those
browser versions, so those codes will never be executed any more.

So in this step, we plan to remove these codes by searching the usage of below
flags in our code library: mstr.utils.ISIE4, ISIE6, ISIE7 and so on.

Risk: Low

Since in our product’s official statement, we don’t support those old browsers
any more, it should be safe to remove related special handling codes.

By using SonarLint [https://www.sonarlint.org/], I find 1514 problems in our JS
codes under mstr folder. Though most of them are not really problems that will
cause issues, but it will make our code much more clean if we solve these

Risk: Low


5.4 Add Unit Test, refactor codes and add necessary comments: 240 points

Since we should not have any plans for new features for prompt, it is too
expensive and risky to re-design the whole framework, so here we will only do
below things:

 1. For the backward compatibility, we should not rename existing methods, so we

refactoring.
