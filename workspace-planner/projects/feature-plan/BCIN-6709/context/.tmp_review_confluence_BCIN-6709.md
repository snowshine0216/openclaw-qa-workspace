# Confluence Review — BCIN-6709

**Domain:** confluence  
**Mode:** review  
**Feature:** BCIN-6709  
**Reviewed Against:**
- `sub_test_cases_atlassian_BCIN-6709.md`
- `qa_plan_atlassian_BCIN-6709.md` (merged Atlassian + Confluence design-doc context)

## Verdict
**PASS WITH GAPS** — the Atlassian sub test cases cover the major Confluence design-doc recovery branches and most of the important business rules. However, several design-doc-specific nonfunctional and precision-coverage expectations are only partially covered or remain too generic.

## What is well covered

### 1. Authoring-only scope guard
Covered well.
- Sub test cases explicitly constrain expected behavior to **Library Web authoring mode**.
- They also explicitly mark **non-authoring / consumption**, **Workstation**, **BI Web**, and **embedding** as out of scope / N/A based on current Atlassian evidence.

### 2. Scenario-specific recovery branches
Covered well.
The design doc says recovery is branch-specific rather than one generic “continue editing” flow. The sub test cases clearly reflect the main branches:
- **Pause mode → Resume Data Retrieval failure**
- **Running-mode manipulation failure**
- **Prompt / reprompt / prompt-in-prompt answer failure**
- **Normal manipulation vs modeling-service manipulation differences**

This is aligned with the merged Atlassian context.

### 3. State restoration expectations
Covered well.
The sub test cases verify that recovery keeps the user:
- in the **same report authoring context**,
- out of **Library home**,
- back in the correct **pause mode** or **prompt state**,
- away from **stale grid / broken editor** states.

### 4. Undo/redo branch semantics
Covered well.
The design doc’s key distinction is represented:
- **normal manipulation crash** preserves undo/redo history,
- **modeling-service failure** resets/clears undo/redo history,
- post-recovery **Undo / Redo / Reset** behavior is explicitly tested.

### 5. Retryability after recovery
Covered well.
The design-doc emphasis on not leaving the UI hanging is reflected by cases checking that:
- retry sends a **fresh request**,
- editor remains responsive,
- user can continue with valid follow-up actions.

### 6. User-visible messaging basics
Mostly covered.
The sub test cases include:
- actionable error guidance for row-limit scenarios,
- differentiated messaging across backend error families,
- prevention of misleading “reopen the report” style recovery messaging.

## Gaps / weaknesses

### 1. Error taxonomy coverage is present but still too generic
**Gap severity: Medium**
The merged Atlassian context calls out concrete backend/error families:
- row-limit exceeded,
- modeling service,
- analytical/query engine,
- SQL,
- MDX/ODBC-style,
- unsupported report type / resource not found / unknown taxonomy paths.

The sub test cases mention several major families, but they stop at a broad validation layer. They do **not clearly ensure coverage of the fuller design-doc taxonomy**, especially:
- unsupported report type,
- resource not found,
- unknown/default mapping behavior.

**Recommendation:** add explicit scenario bullets validating fallback/error-string mapping for unsupported / resource-not-found / unknown taxonomy branches.

### 2. Recreate-instance gating is only covered as automation, not user-visible validation
**Gap severity: Medium**
The design doc makes an explicit business rule / assumption:
- **recreate server instance only when certain manipulations crash the instance**.

The sub test cases mention this only under **AUTO: Automation-Only Tests**:
- “Recreate-instance gating”

That means the rule is acknowledged, but there is **no clear functional black-box coverage** that distinguishes:
- crashed-instance path requiring recreation,
- non-crashed or modeling-service path not requiring the same recovery side effect.

**Recommendation:** add a functional scenario comparing a crashed manipulation path vs a non-recreate path, validating the resulting user-visible recovery branch and state continuity.

### 3. Prompt dependency / assumption around F43454 is not directly guarded
**Gap severity: Medium**
The merged Atlassian context explicitly states that the prompt-answer recovery for **maximum row limit exceeded** depends on:
- **F43454 cancel prompt answer and back to prompt**.

Current sub test cases cover returning to prompt and preserving answers, but they do **not explicitly guard this dependency/assumption**.

**Recommendation:** add a scenario that verifies cancel/back-to-prompt behavior remains compatible with the BCIN-6709 recovery path for row-limit failures.

### 4. Performance coverage is intentionally present but still placeholder-level
**Gap severity: Medium**
The sub test cases include performance placeholders:
- acceptable normal recovery time,
- large-data recovery without obvious long hang,
- TODO notes saying no concrete SLA/threshold exists.

This is acceptable as a placeholder, but from a review standpoint the design-doc nonfunctional coverage is **not yet operationalized**.

**Recommendation:** mark these as provisional and either:
- attach product-default timing expectations, or
- defer to a dedicated performance plan with explicit thresholds.

### 5. Accessibility/UX resilience coverage is thin relative to recovery complexity
**Gap severity: Low-Medium**
There are useful P3 checks for:
- keyboard-only popup dismissal,
- focus returning to usable location,
- no keyboard trap,
- visual stability during recovery.

However, given the design’s heavy use of prompt overlays, error popups, and branch transitions, the coverage is still somewhat thin. Missing more explicit checks include:
- focus restoration to the **specific prior prompt field/control** when returning to prompt,
- screen-reader-friendly recovery messaging behavior,
- keyboard navigation across nested prompt-in-prompt recovery after failure.

**Recommendation:** expand accessibility coverage for prompt recovery and nested modal/overlay behavior.

### 6. Messaging validation does not fully prove per-type string mapping fidelity
**Gap severity: Low-Medium**
The sub test cases check that messages differ appropriately and provide guidance, but they do not enumerate a stronger mapping verification for each major error-type bucket surfaced in the design doc.

**Recommendation:** add a compact matrix verifying representative message title/body mapping for each major backend error family and default/fallback bucket.

## Recommended additions before refactor/final synthesis
1. Add explicit error-taxonomy scenarios for:
   - unsupported report type,
   - resource not found,
   - unknown/default fallback mapping.
2. Add a functional comparison for **recreate-instance required vs not required** recovery branches.
3. Add an explicit row-limit dependency check tied to **F43454 cancel/back-to-prompt** behavior.
4. Strengthen accessibility around:
   - prompt focus restoration,
   - nested prompt keyboard flow,
   - screen-reader-friendly recovery messaging.
5. Convert current performance placeholders into either measurable expectations or an explicit deferred-performance handoff.
6. Add a lightweight message-mapping matrix proving visible string behavior per major error family.

## Final assessment
The Atlassian sub test cases are **substantively aligned** with the Confluence design doc and already capture the most important business-rule branches: authoring-only scope, branch-specific recovery, state preservation, retryability, prompt recovery, and undo/redo differentiation.

The main remaining issues are not missing core flows, but **precision and nonfunctional completeness**: fuller error-taxonomy mapping, stronger recreate-instance gating validation, explicit dependency guarding for the row-limit prompt path, and more concrete accessibility/performance expectations.
