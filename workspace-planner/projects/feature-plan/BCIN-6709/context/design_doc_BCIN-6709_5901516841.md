## Testing-relevant takeaways

### Explicit scope / assumptions
- Scope is **report authoring mode only**; consumption mode is out of scope.
- Feature only recreates server instance when **certain manipulations crash the instance**.
- Prompt-answer error for **maximum row limit exceeded** depends on **F43454 cancel prompt answer and back to prompt**.

### Recovery branches called out by the design
1. **Pause mode → Resume Data Retrieval → rebuildDocument error with crashed instance**
   - Return to pause mode.
   - Preserve prior manipulations when applicable.
   - Clear error guidance should be shown.

2. **Running-mode manipulation error with crashed instance**
   - Recover document view correctly instead of leaving stale/broken grid state.
   - Distinguish normal manipulation vs modeling-service manipulation behavior.

3. **Prompt / reprompt / prompt-in-prompt answer apply error (Library Web)**
   - Return to prompt.
   - Preserve previous prompt answers.
   - Keep previous mode where applicable.

### Undo/redo expectations
- **Normal manipulation crash**: keep undo/redo history after recreate-instance recovery.
- **Modeling-service manipulation error**: clear/reset undo/redo history.
- Dedicated edge case exists for undoing a crashed manipulation so report objects/UI become removable/consistent again.

### View/state correctness expectations
- Do not force navigation to Library home.
- Do not leave document view stuck in stale grid view when pause-mode empty view is expected.
- Do not leave empty/broken report-editor grid after recovery.
- Retry / follow-up manipulations must still send requests instead of hanging.

### Error families concretely referenced
- Maximum row limit exceeded.
- Modeling service manipulation errors.
- Analytical engine / query engine failures.
- SQL failure.
- MDX / ODBC-style backend failures.
- Prompt answer apply failures.

### User-visible messaging implications
- Error titles/messages vary by type.
- Row-limit failure includes actionable guidance.
- Prompt-answer failures in Library Web have specific transformed handling.

## Scope impact assessment
Yes — this design doc materially expands testcase scope from a generic "continue editing after error" theme into explicit scenario families with different expected recovery states, undo/redo rules, and prompt-handling behavior.
