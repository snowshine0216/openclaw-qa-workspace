# Site Knowledge Output Alignment Fix Plan

> **Plan ID:** `site-knowledge-output-alignment-v1`
> **Date:** 2026-03-04
> **Status:** Design Only (No Implementation Changes)
> **Scope:** Align expected output contract in `TESTER_AGENT_DESIGN_v2.md` with actual generated files in `memory/site-knowledge/`

---

## 1. Problem Statement

There is a contract drift between:

1. Expected output described in [TESTER_AGENT_DESIGN_v2.md](./TESTER_AGENT_DESIGN_v2.md) section `2.5` and `2.6`
2. Current output format generated by `tools/sitemap-generator`
3. Current system design assumptions in [SITE_KNOWLEDGE_SYSTEM_DESIGN.md](./SITE_KNOWLEDGE_SYSTEM_DESIGN.md)

### 1.1 Observed Gap (Expected vs Actual)

| Area | Expected (Tester Design v2) | Actual (`memory/site-knowledge`) | Gap |
|------|------------------------------|----------------------------------|-----|
| Layer 1 `SITEMAP.md` | Narrative "Product Areas" sections with navigation path, key UI entry points, components, and `query sitemap:<domain>` hints | Flat domain index (`## <domain>`, component count, detail file) | Missing semantic/navigation content |
| Layer 2 `<domain>.md` | Human-facing "Overview + Components + User-visible elements + Key actions + Related components" | Technical locator/action tables derived from raw POM parse | Shape mismatch with expected agent-readable narrative |
| Contract boundary | `PomSummary` is internal extraction format | Internal structure leaks into external-facing markdown contract | External format is implementation-driven instead of contract-driven |
| Test coverage | Should enforce "expected output equals generated output" | Current tests validate mostly existence/contains and are stub-oriented | No golden-contract verification |

### 1.2 Why This Matters

1. The Tester Agent consumes these files as runtime knowledge.
2. If output shape differs from design expectation, prompt instructions and retrieval behavior become unstable.
3. Without strict contract tests, future refactors can silently reintroduce drift.

---

## 2. Target Contract (Canonical)

This plan sets one canonical rule:

1. `TESTER_AGENT_DESIGN_v2.md` section `2.5` and `2.6` define the user-facing markdown contract.
2. `PomSummary` remains an internal intermediate object only.
3. Generated markdown artifacts must match the contract examples semantically and structurally.

### 2.1 Layer Contract

| Layer | File | Audience | Contract |
|-------|------|----------|----------|
| L0 (internal) | `PomSummary` in memory only | Generator internals | Structured parsed POM data |
| L1 (always available) | `SITEMAP.md` | Tester Agent startup context | Product-area narrative format from section `2.6` |
| L2 (domain detail) | `<domain>.md` | Domain deep-dive lookup | Domain knowledge sheet format from section `2.5` |
| L3 (metadata) | `metadata.json` | Traceability/debugging | Generation timestamp, source, component counts, detail file paths |

### 2.2 Output Invariants

1. `SITEMAP.md` includes all in-scope domains in deterministic order.
2. Every domain listed in `SITEMAP.md` has a corresponding `<domain>.md`.
3. Each `<domain>.md` follows the section order defined in `2.5`.
4. Internal `PomSummary` fields are transformed into narrative output; they are not dumped as raw internal schema.

## 3. Remediation Plan (Design-Level)

### Phase A: Contract Lock (Docs First)

1. Declare canonical external output schema in `SITE_KNOWLEDGE_SYSTEM_DESIGN.md`.
2. Add explicit "Current vs Target" note so readers do not treat current output as compliant.
3. Add examples for one domain (`filter`) and one compact sitemap segment as canonical fixtures.

Deliverable:
- Updated design docs with explicit canonical output contract and migration note.

### Phase B: Generator Alignment (Future Implementation)

1. Refactor renderers to map internal `PomSummary` to narrative sections.
2. Keep parser extraction logic separate from renderer formatting logic.
3. Add deterministic ordering rules:
   domain order in `SITEMAP.md`, component order within each `<domain>.md`,
   and locator/action ordering for stable output.
4. Preserve `metadata.json` for machine verification.

Deliverable:
- Implementation PR aligning renderer outputs with canonical contract.

### Phase C: Contract Validation (Minimal Mock Policy)

1. Add fixture-driven golden tests for final markdown outputs.
2. Add end-to-end generation test from local fixture repository.
3. Fail tests on any structural or wording drift that violates canonical templates.

Deliverable:
- Test suite that proves expected output equals generated output.

## 4. Test Strategy (Expected == Actual, Minimum Mock)

### 4.1 Mocking Policy

Use minimum mock by default:

1. No mocks for pure render functions (`buildDomainSheet`, `buildCompactSitemap`).
2. No mocks for filesystem flow in local integration tests (use temp dirs + real fixture files).
3. Only mock boundary cases: clock/timestamp provider (for deterministic snapshots)
   and remote GitHub API calls in `github.mjs` tests.

### 4.2 Required Test Layers

1. Unit contract tests (pure functions):
   verify exact markdown structure and section headers, and verify deterministic ordering.

2. Golden file tests (fixture-based):
   use fixed POM fixtures and compare exact `SITEMAP.md`/`filter.md` to committed expected files.

3. End-to-end local test (no network):
   run CLI on fixture repo and assert generated artifacts match golden files byte-for-byte
   (except normalized timestamp placeholder).

4. Cross-doc contract check:
   validate design-doc examples stay synchronized with golden fixtures and prevent drift
   between `TESTER_AGENT_DESIGN_v2.md` and system design docs.

### 4.3 Pass Criteria

All criteria must pass:

1. Generated `SITEMAP.md` matches canonical Layer 1 template.
2. Generated `<domain>.md` matches canonical Layer 2 template.
3. Coverage includes at least one multi-component domain and one edge case domain.
4. Zero snapshot drift unless docs and golden fixtures are intentionally updated together.

## 5. Risks and Controls

| Risk | Impact | Control |
|------|--------|---------|
| Narrative rendering introduces ambiguous heuristics | Inconsistent summaries across domains | Define deterministic mapping rules and fallback wording templates |
| Timestamp/source lines break snapshots | Flaky tests | Inject clock/source in tests; normalize dynamic fields |
| Legacy output consumers rely on old format | Runtime regressions | Add migration note and compatibility window in implementation PR |
| Over-mocking hides real drift | False confidence | Enforce minimum-mock policy and end-to-end fixture runs |
